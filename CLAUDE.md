## 1. What This Is

**SPT-Txn** — Scoped, Provable, Transaction-bound authorization tokens.

The problem: current authorization is *role-scoped*. An actor (human, workload, or AI agent) is granted a role, and that role's authority persists across every action it takes. This fails exactly when actors fail — under compromise, under prompt injection, under goal hijacking — because a compromised actor retains full role authority.

SPT-Txn is *transaction-scoped*. Authority exists only inside a short-lived token bound to **one declared action**, on **one resource**, under **one jurisdictional policy**, verified against **what the actor is and how it was attested**. A compromised agent holds a token that is cryptographically useless for any action other than the one it declared.

Every decision emits a signed, tamper-evident receipt as a byproduct. Compliance evidence is not generated by an audit; it is generated by enforcement.

**Existing assets (all public, all citable):**
- IETF draft: `draft-coetzee-oauth-spt-txn-tokens-04`
- Formal cryptographic theory paper with five game-based security proofs
- Zenodo DOI `10.5281/zenodo.19299787`; SSRN Abstract ID `6379940`
- ORCID `0009-0009-6557-8843`
- Go PoC: Ed25519 / JWT / EdDSA issuance, format-agnostic TBAC engine (CIMA/VARA/SEC), strict-mode enforcement

**Why this matters commercially:** NIST/CAISI and NCCoE are actively standardizing AI agent identity and authorization *right now*. The construct they are converging toward is this one. Standards position is the asset; the code is the credential that makes the standards position believable. **Spec text that lands in the right body is worth more than any amount of polished implementation.** Prioritize accordingly when the two compete for time.

## 2. Security Invariants — Non-Negotiable

These are enforced, not aspirational. Code that violates them does not ship, regardless of how well it works.

**Deny by default, fail closed.** No PEP, adapter, or gateway ever fails open. Policy engine timeout, malformed token, unreachable revocation source, provenance evidence unavailable — all resolve to DENY *with an evidence record explaining why*. There must be a distinguishable decision class for "denied due to violation" vs. "denied due to unavailability", because operators need to tell an outage from an attack.

**Enforce structurally, not by convention.** Deny-by-default must be a property of the type system: it should be *impossible to construct* a request object that has passed through the PEP without a decision attached. Do not rely on a runtime `if` that a future refactor can delete. If a reviewer can silently remove a check and the code still compiles, the design is wrong.

**No ambient authority.** Nothing is ever authorized because of where it sits on a network, what long-lived secret it holds, or what role it was assigned. Authority lives only in a short-lived, transaction-scoped, cryptographically bound token. If you catch yourself writing "if the caller is on the internal network" or "if the API key is valid, allow", you have violated the core thesis of the product.

**Attenuation is monotonic and cryptographic.** Any delegation step can only *narrow* authority, never widen it. This must be enforceable **offline**, by construction, without a call home. A policy convention that says "children should be narrower" is not attenuation. Child TTL must be strictly less than parent TTL. Delegation depth is bounded and the bound fails closed.

**Evidence is a byproduct, never an afterthought.** Every authorization decision emits a signed record at the moment of decision. If a component cannot produce evidence, it does not ship. Receipts go to an append-only Merkle transparency log; nobody — including us and including the customer operator — can silently rewrite history.

**No custom cryptography. Ever.** Standard library and audited implementations only (`crypto/ed25519`, `crypto/subtle`, audited PQC libs). **If any model or contributor proposes hand-rolling a primitive, a construction, a padding scheme, or a canonicalization from scratch, that is an automatic reject with no discussion.** This is the single most reliable source of catastrophic bugs in security products.

**Constant-time comparison for anything secret-adjacent.** Tokens, MACs, signatures, digests. `crypto/subtle.ConstantTimeCompare`. A `==` on a MAC is a vulnerability, not a style issue.

**Memory-safe languages only in the trust boundary.** Go (current) or Rust (acceptable for the latency-critical data-plane sidecar). **No C or C++ anywhere inside the trust boundary**, including via cgo. If a dependency pulls in cgo for crypto, that dependency is rejected.

**Keys never live in application memory longer than necessary.** Issuer keys in HSM/KMS, or at minimum TPM/keychain-backed. Key rotation and *algorithm* rotation are both designed-in operations with overlap windows — not incident procedures. Log signing key is separate from token issuance key, with a separate rotation schedule.

**Latency is a security requirement.** The decision path must hold p99 under ~10ms. This is not a performance nicety: a PEP that is too slow gets bypassed by platform teams, and a bypassed PEP is *worse than no PEP*, because it creates the false belief that enforcement exists.

**We eat our own supply chain.** Signed commits, pinned dependencies, `govulncheck` in CI, SLSA provenance on releases, published SBOM. We sell attestation-conditioned authorization; an unattested build pipeline is disqualifying and any acquirer's technical DD will notice.

## 3. Threat Model Summary

The full STRIDE-per-component analysis is in `docs/THREAT-MODEL.md`. Read it before touching trust-boundary code. The bugs that will actually kill this product, in likelihood order:

1. **Canonicalization mismatch.** The verifier canonicalizes a request differently than the issuer did when it computed the intent digest. Result: full authorization bypass. This is *the* classic bug class in this design. The canonicalization scheme must be specified exactly (JCS / RFC 8785), implemented once, shared by both paths, and fuzzed against itself.
2. **Attenuation bypass.** Child token validated in isolation without verifying the full parent chain, or a caveat that widens rather than narrows. Result: privilege escalation.
3. **Downgrade attack on algorithm agility.** Attacker forces the weaker suite in hybrid mode. The negotiated suite must itself be covered by the signature, and policy floors must be non-bypassable.
4. **Replay.** Transaction-scoped tokens replayed within their (short) validity window. Nonce binding, single-use enforcement at the PEP.
5. **Parser vulnerabilities.** Malformed token handling. Fuzz it.
6. **Revocation gap.** Chain depth > 1 without a status check is a window of unrevokable authority.
7. **Confused deputy at the PEP.** The PEP forwards upstream credentials or acts with its own authority instead of the token's. (This is precisely the MCP token-passthrough gap — do not reproduce it in our own middleware.)

## 4. Compliance & Standards Targets

Write code as if it will be read by a bank's security architecture review board and a federal ATO package reviewer, because it will be.

- **NIST SP 800-53** control mappings maintained per component
- **FIPS 140-3** validated crypto modules where the deployment requires it — design so the crypto backend is swappable without touching call sites
- **CNSA 2.0 / PQC**: hybrid signing (classical + ML-DSA), verify-either in transition, verify-both in strict mode
- **NIST AI RMF / CAISI agentic controls**: full decision-chain audit for agent actions
- **CISA ZTMM**: identity, applications & workloads pillars
- EU: **DORA, NIS2, CRA, MiCA** — jurisdictional TBAC profiles
- **OWASP Top 10 for Agentic Applications** — goal hijacking (ASI01) is directly addressed by intent binding

## 5. How to Work in This Repo

### Model selection

**Opus for the trust boundary. Sonnet for everything else.**

*Trust boundary* (≈15% of code, ≈95% of risk) — Opus, slowly, spec-first:
- token issuance, signing, envelope construction
- signature verification and canonicalization
- delegation chain attenuation
- TBAC policy evaluation and all fail-closed paths
- key handling, rotation, HSM/KMS integration
- hybrid PQC signing and suite negotiation
- Merkle log append and inclusion proofs

*Everything else* — Sonnet is fine: adapters, Envoy filter plumbing, OPA shim, CLI, test harnesses, CI, docs, demo.

### Mandatory review loop for trust-boundary code

Never single-pass. The order is:

1. **Spec and threat model first.** No code until the invariant being enforced is written down.
2. **Implement against the spec.**
3. **Adversarial review in a FRESH context.** The brief is *"Assume this code contains an authorization bypass. Find it."* — **not** "review this code," which produces agreement rather than analysis. A model asked to *find* a flaw behaves very differently from one asked to *check* for flaws. This distinction is load-bearing; do not soften the prompt.
4. **Property-based tests.** See below.
5. **Human line-by-line review.** The maintainer holds CISSP-ISSAP / ISSMP / CSSLP. This is the step he is actually credentialed for and it is not delegable to a model.

### Testing requirements

- **Property-based testing on attenuation is the highest-value test in the codebase.** Generate random delegation chains; assert authority *never widens* at any hop, under any input. If you write one test, write this one.
- **Differential test the canonicalizer against itself** across issuer and verifier paths.
- **Fuzz the token parser.** Every malformed-input path.
- Negative tests for every fail-closed path — assert DENY, and assert the evidence record explains *why*.
- `govulncheck` in CI, blocking.

### Working style

- Small, reviewable commits. Trust-boundary changes never bundled with unrelated refactors.
- Trust boundary lives in its own package with a deliberately narrow public interface, so "code requiring Opus + adversarial review" is a *directory*, not a discipline someone has to remember.
- When a task is ambiguous, ask. Guessing in an authorization engine is how bypasses get written.

## 6. Roadmap Context

Full version in `docs/ROADMAP.md`. Current priority order:

1. **Agentic delegation + intent binding + MCP PEP profile** — hardest deadline. NIST/NCCoE reference architectures are being scoped now. If anything slips to protect this, let it slip.
