SPT-Txn · Scaling & Deployment · Violet Sky Security SEZC

No central runtime. It scales like JWT verification.

SPT-Txn verifies offline, inside the adopter's own infrastructure — the presented tokens plus a locally-cached Trust Registry snapshot, and nothing else. There is no call back to any SPT-Txn server and nothing of ours in the transaction path. The reference OpenBSD host is an issuer and a demo, not the runtime. Each verification is a stateless pure function, so throughput scales horizontally with cores — the profile large platforms already run for JWTs.

Cleartext verify
~62k/s
Full 3-hop chain verifications/sec on a 10-core laptop; scales linearly with cores. Signature + hash, fails closed.
Token minting
~394k/s
Ed25519 action-token issuance/sec (10 cores). Per-participant — each adopter mints from its own infra.
Privacy path · ZK
~1.5 ms
Per Groth16 N-hop verify (~6.7k/s, 10 cores). Proving (~159 ms) is once per chain, amortized over every action.

What an adopter integrates

  • Verifier library — embedded in-process; the whole eight-step check runs locally.
  • Registry snapshot feed — a signed JSON pulled and cached periodically (like JWKS/CRL).
  • Their own issuer — only if they mint tokens; otherwise nothing to run.
  • Optional on-chain anchor — a periodic registry-root write for tamper-evidence.

Why there is no bottleneck

  • Stateless verify — a pure function of (tokens, snapshot); add replicas, add throughput.
  • No live issuer/chain call in the hot path; registry & revocation are cached snapshots.
  • ZK proving is off the hot path — once per delegation chain, parallelizable.
  • One small stateful piece — the DPoP anti-replay cache, TTL-bounded and shardable by holder.

Portable & FIPS-ready. The reference host is hardened OpenBSD (pledge/unveil) — a security choice, not a requirement. The same Go binary runs in a FIPS 140-3 environment: enable Go's native FIPS mode on a FIPS-validated Linux (RHEL / AlmaLinux / Rocky) so classical crypto runs through the CMVP-validated Go Cryptographic Module (cert #5247), or route issuer keys to a FIPS 140-3 HSM over PKCS#11 (signing already uses crypto.Signer — a config swap). The zero-knowledge layer is outside FIPS by nature — a distinct, opt-in component.

Honest scope: benchmarked and live across many chains is not the same as production load at a single adopter — that is what a design-partner pilot establishes. What is de-risked by design is the architecture: stateless, offline, horizontal, with no central runtime — the same shape as the JWKS / CRL / JWT verification the industry already runs at scale.
